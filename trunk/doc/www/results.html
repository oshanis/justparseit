Results


We have fully implemented the Policy Parser and deployed it on the web. The following is a list of example sentences from which our system is able to successfully generate RDFs:


  • Police may search people's homes if police have people's permission
  • MIT may use prox card data for criminal investigation
  • MIT students can not transfer prox cards to another student
  • A person may access data on the private mit domain if the person has permission for that data
  • A service provider may not use phone records to deny service to a customer
  • TSA can transfer data to the FBI if the data is associated with terrorism
  • Anyone can access data in the possession of the state of massachusetts


This represents only a small subset of all sentences that our system is able to handle. For example, since our underlying CFG uses spelling change automata, it can successfully parse all sentences with variations that are recognized by these automata - for example, adding "s" to pluralize a noun or recognizing that 1st-person and 3rd-person verbs with the same root. In addition, by using the features in the CFG, the system is able to interpret whether two different sentences have the same meaning. For example, the following two sentences result in the generation of the same RDF:


  • MIT can give records to FBI to investigate crimes.
  • MIT can give FBI records for criminal investigations.


The capacity of the Parser depends on three parameters. The first is the FCFG (available here), which determines the structure of sentences that the Parser is able to parse. As discussed in the Implementation page, the current FCFG recognizes the sentences of the form


subject [mod] action object ['to' secondary_object] ['for' purpose] [condition]*


The second parameter is the lexicon and the spelling change rules, both of which are used by the FCFG as terminals. Technically, the number of sentences that are recognized by the FCFG is infinite, since the grammar is recursive (for example, it can parse sentences with an arbitrary number of repetitive propositional phrases such as "Committee on Discipline on Discipline on Discipline..."). However, only a finite subset of this infinite set contains meaningful sentences, so the capacity of our system cannot be judged purely by the number of sentences that it parses.


The third parameter that determines the capacity of the Policy Parser is the set of domain ontologies (combined into a single file), which act as vocabularies during RDF generation. Therefore, even if the system is able to parse (NLTK parser) and interpret (Policy Interpreter) an input policy, it cannot generate a corresponding RDF if the constituents of the policy cannot be mapped to corresponding components of an ontology.


All these three parameters together define the limitation of our system, but this also means that each of them can be extended in order to make the system more flexible. For example, the FCFG can be modified to handle a wider variety of sentence structures; the lexicon can be expanded; and new ontologies can be added generate more RDFs. These possible future extensions are discussed more thoroughly in the Future Directions section.


Another limitation of our system is that it cannot relate pronouns to its corresponding noun phrase. For example, the sentence


A student can access data on the private MIT domain if she has the permission for that data


is rejected by the Policy Parser; to correct this, the user must replace "she" in the "if" condition phrase with "the student". In this particular case, it would have been possible to infer that the pronoun "she" actually refers to "the student", but in some other cases, such inferences can run into ambiguities that must be resolved - this is out of the scope of this project, but is considered a possible future extension.
